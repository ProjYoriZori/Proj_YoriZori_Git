## 1. 테스트 범위 및 전략

### 1.1 테스트 목표

- 사용자가 앱을 처음 실행하는 순간부터 로그아웃까지 전체 흐름을 검증
- 각 기능이 MVP 기준에 부합하는지 확인
- 공공데이터 API(식품의약품안전처) 연동 안정성 검증
- 경계값·예외 케이스를 통해 서비스 품질 사전 확보

### 1.2 우선순위 정의

| 등급 | 설명 |
| ---- | ---- |
| P1 | 서비스 핵심 기능 — 반드시 통과해야 배포 가능 |
| P2 | 주요 기능 — 릴리즈 전 완료 권장 |
| P3 | 부가 기능 — 고도화 단계에서 검증 |

### 1.3 테스트 환경

| 항목 | 값 |
| ---- | -- |
| OS | Windows 11, macOS Sonoma |
| 브라우저 | Chrome 최신, Safari 최신, Edge 최신 |
| 해상도 | 1920×1080(데스크톱), 390×844(모바일 에뮬레이션) |
| 백엔드 | Spring Boot, MySQL 8.x |
| API | 식품의약품안전처 조리식품 레시피 DB |
| 테스트 DB | 로컬 yorijori MySQL 인스턴스 |

## 시나리오 01 — 앱 실행

> **목표**: 서비스 최초 진입 시 초기 화면이 정상 렌더링되고, 필수 리소스(레시피 데이터, 이미지)가 로드되는지 확인한다.

| 테스트 ID | 시나리오 | 테스트 목적 | 사전 조건 | 입력값 | 수행 절차 | 예상 결과 | 우선순위 |
| --------- | -------- | ----------- | --------- | ------ | --------- | --------- | -------- |
| TC-001 | 앱 최초 진입 | 홈 화면이 정상 렌더링된다 | 서버 정상 기동, DB에 레시피 데이터 최소 1건 이상 적재 | 브라우저에서 서비스 URL 입력 | 1. 브라우저에서 `http://localhost:3000` 접속 | 홈 화면이 2초 이내 렌더링되고 하단 탭 네비게이션이 표시된다 | P1 |
| TC-002 | 공공 API 데이터 로딩 | 레시피 DB 데이터가 화면에 노출된다 | 식품의약품안전처 API 정상 연동 상태 | 없음 | 1. 홈 화면 진입 후 레시피 카드 영역 확인 | 공공데이터 기반 레시피가 1건 이상 노출된다 | P1 |
| TC-003 | API 서버 다운 상태 진입 | 서버 오류 시 사용자 친화적 오류 메시지가 표시된다 | 백엔드 서버 중지 상태 | 없음 | 1. 서버 중지 후 접속 | "서비스에 일시적 문제가 발생했습니다" 안내 문구가 표시된다 | P1 |
| TC-004 | 네트워크 없는 환경 진입 | 오프라인 상태를 감지하고 안내한다 | 네트워크 연결 없음 | 없음 | 1. 개발자도구 → Network → Offline 설정 후 접속 | "네트워크 연결을 확인해 주세요" 메시지가 표시된다 | P2 |
| TC-005 | 로딩 속도 검증 | 첫 화면이 3초 이내 렌더링된다 | 정상 환경, 레시피 20건 적재 | 없음 | 1. Network 탭에서 로딩 시간 측정 | First Contentful Paint가 3초 이내 완료된다 | P2 |

## 시나리오 04 — 메인 화면 진입

> **목표**: 로그인 후 홈 화면의 각 구성 요소가 정상 렌더링되고, 보유 식재료 미등록 시 안내 문구가 표시되는지 확인한다.

| 테스트 ID | 시나리오 | 테스트 목적 | 사전 조건 | 입력값 | 수행 절차 | 예상 결과 | 우선순위 |

| TC-024 | 보유 식재료 등록 후 홈 진입 | 선택된 재료 카드가 홈에 표시된다 | 냉장고에 식재료 3건 이상 등록 | 없음 | 1. 냉장고에서 재료 선택 2. 홈 화면 복귀 | 선택된 재료 카드가 홈 화면에 표시 | P1 |

| TC-028 | 제철 식재료 배너 표시 | 홈 상단에 제철 식재료 배너가 노출된다 | 제철 식재료 데이터 존재 | 없음 | 1. 홈 화면 상단 확인 | 현재 월에 해당하는 제철 식재료 배너 표시 | P3 |
